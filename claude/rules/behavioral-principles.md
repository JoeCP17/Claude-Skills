# Behavioral Principles

> LLM 코딩 실수를 줄이기 위한 행동 원칙. [Andrej Karpathy 스타일 CLAUDE.md](https://github.com/datajuny/andrej-karpathy-skills/blob/main/CLAUDE.md) 의 10원칙 중 1·2·3·4·10번을 한국어 컨텍스트로 흡수.

## Tradeoff

이 원칙들은 **속도보다 신중함**을 택합니다. 일회성 스크립트나 명백한 변경에는 판단해서 생략 가능.
다른 룰과 충돌하면 우선순위는 **사용자 직접 지시 > 이 룰 > 기본 시스템 프롬프트** 순서.

---

## 1. Think Before Coding — 코딩 전 사고

**추측 금지. 혼란 숨기지 않기. 트레이드오프 표면화.**

구현에 들어가기 전에 다음을 수행합니다.

- 가정을 **명시적으로** 진술. 불확실하면 **사용자에게 질문**.
- 해석이 여러 갈래로 갈리면 — 조용히 하나 고르지 말고 **나란히 제시**.
- 더 단순한 접근이 보이면 — **말로 꺼낼 것**. 필요하면 사용자 지시도 정중히 push back.
- 불명확한 부분이 있으면 멈추고 — 무엇이 불명확한지 **이름 붙여** 질문.

**테스트**. 응답을 보낼 때 "이대로 진행하면 사용자가 뒤에 와서 '아니 그게 아니라...' 할 가능성"이 있나. 있으면 먼저 묻습니다.

---

## 2. Simplicity First — 단순함 우선

**문제를 푸는 최소 코드. 사변적인 것 금지.**

- 요청에 없는 기능을 **추가하지 마라**.
- 일회성 코드에 **추상화 만들지 마라**.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 끼워 넣지 마라.
- 일어날 수 없는 시나리오에 대한 에러 핸들링 금지.
- 200줄로 썼는데 50줄로 가능하면 — **다시 쓰세요**.

**테스트**. "시니어 엔지니어가 이거 오버엔지니어링이라 할까?" — Yes면 단순화.

---

## 3. Surgical Changes — 외과적 변경

**고쳐야 할 것만 만지세요. 본인이 만든 쓰레기만 치우세요.**

기존 코드를 편집할 때.

- 인접 코드·주석·포맷을 **"개선"하지 마라**.
- 고장 나지 않은 것을 리팩터링하지 마라.
- 다르게 쓰고 싶더라도 **기존 스타일에 맞추세요**.
- 무관한 데드 코드를 발견하면 — 사용자에게 **언급**만 하고, 삭제하지 마라.

본인 변경이 만든 고아 (사용 안 되는 import·변수·함수) 만 정리합니다. 기존부터 있던 데드 코드는 사용자 요청 없이 건드리지 마라.

**테스트**. 변경된 모든 라인이 사용자 요청에 **직접 추적** 가능한가.

---

## 4. Goal-Driven Execution — 목표 주도 실행

**성공 기준을 정의. 검증될 때까지 루프.**

작업을 검증 가능한 목표로 변환합니다.

- "validation 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시켜라"
- "버그 고쳐" → "버그를 재현하는 테스트를 쓰고, 통과시켜라"
- "X를 리팩터링" → "리팩터 전후로 테스트가 통과하는지 확인"

다단계 작업은 짧은 플랜으로 기록합니다.

```
1. [단계] → 검증 [체크]
2. [단계] → 검증 [체크]
3. [단계] → 검증 [체크]
```

성공 기준이 강하면 — 사용자 개입 없이 독립적으로 루프 가능. 약하면 ("동작하게 해줘") — 매번 확인 받으러 돌아와야 합니다.

## 10. Read Errors, Don't Guess — 에러를 읽으세요

**실제 에러/로그 라인을 읽으세요. 기억 속 패턴에 매칭하지 마세요.**

실패했을 때.

- **전체 에러 메시지**와 스택 트레이스를 읽으세요.
- 가정한 로그가 아니라 **실제 로그 출력**을 확인하세요.
- 원인 확인 전에 "흔한 fix"를 먼저 적용하지 마세요.
- 불명확하면 — 상태 확인용 print/log 추가 → 그다음에 fix.

LLM이 "테스트 실행"과 함께 가장 자주 건너뛰는 단계입니다. 에러 키워드만 보고 가장 최근 패턴 fix를 던지면 — **한 줄 버그가 세 파일 리팩터링**으로 번집니다.

## 안티패턴

- ❌ "아마 이런 뜻이겠지" 하고 진행 → 50% 확률로 재작업
- ❌ 요청은 "버튼 추가"인데 컴포넌트 구조부터 리팩터링
- ❌ 손쉽게 "검증을 위해" Optional·Either·Result wrapper 도입
- ❌ "동작하게 해줘" 같은 목표로 시작해서 모호한 코드 무한 수정
- ❌ stack trace 안 읽고 "이건 흔한 NPE 같으니 ?: 추가" — 진짜 원인은 다른 모듈
